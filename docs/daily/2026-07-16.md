## ★教訓

1. **前任の「評価」は根拠を確認してから採用する。** 「型アーカイブは off-brand」を自分で見ずに推奨へ載せかけた。実測したら**機能は実在し動いていた**。off-brand という結論自体は正しかったが、**理由は私が思っていたものと違った**（CMS名の漏れではなく、bespoke org には意匠を載せる先が無い）。数字で確かめて初めて正しい理由に辿り着いた。
2. **🔴 素の curl は公開サイトについて嘘をつく。** NENE2 はコンテントネゴシエーションするので、curl 既定の `Accept: */*` には **API の JSON** を返す。結果 **`/` が壊れて見え、ソフト404が実404に見えた**。**公開面の検証には必ずブラウザ相当の Accept を付ける**こと。さらに **200 だけでは何も証明できない**（SPAシェルは任意のパスに 200 を返す）。`<title>` が `NeNe Records Admin` ならソフト404。
3. **「数字が0件」は「直った」ではない。** デプロイ直後の検証で「`Record #<id>` が0件＝直った」と出たが、実際は画面が **Loading のまま何も描画されていなかっただけ**。スクリーンショットを見て気づいた。
4. **計測手段は4回壊れた（curl の Accept ×2／待機条件／Vite の allowedHosts が返す403ページを「SPA が起動しない」と誤読）。毎回スクリーンショットか本文を見て捕まえた。** セレクタの数字だけ見ていたら全部誤報告になっていた。
5. **モックが API と同形でないと、契約違反はテストをすり抜ける。** #891 の本体は1行だが、**根本は msw に `meta_title` が無かったこと**。#849→#875→#891 と3度同じ双子を繰り返した理由がこれ。個別に気づく運用は破綻している。
6. **自分が書いた制約が、自分の設計を殺すことがある。** 依頼書の「セレクタは必ず `.nene-public` から」が、**A群に限って正反対**だった（bare は `.nene-public` を持たない）。ClaudeDesign は指示に忠実だったので、**そのままなら AYANE で一切効かなかった**。
