### B. 🔴 公開SSR/SPA のちらつき — 独立した原因が4つ

| Issue / PR | 原因 |
|---|---|
| **#879 / #880** | **SSR が `layout` を無視**し `bare` にも汎用 chrome（パンくず・`← ページ`・`<h2>content</h2>`）を出力。**`PublicLayouts::resolve()` は実装もテストも存在したが、呼んでいるのは自身のテストだけ**＝配線漏れ。docblock 自身が `bare: no header/footer, no theme` と書いており**実装が仕様に違反**していた |
| **#881 / #882** | SPA が bootstrap から **permalink 解決を seed していない** → resolve 待ちで `RecordShellMessage`（テーマ chrome＋Loading…）を描画。bootstrap に `canonicalPath` を追加して seed |
| **#883 / #884** | ①**1セグメント URL は SPA が「タイプ一覧」と解釈**し `PublicBrowsePage` に入る。その型一覧クエリが seed と**キー不一致**（`{limit:20}` vs `{limit:100}`）②**blocks だけ seed 不能で毎回フェッチ**し `isLoading` を握っていた（サーバは型が宣言するデータ型しか読まない設計なのに **SPA だけ無条件**） |
| **#887 / #888** | **bootstrap の `entityTypes` が `{id,name,slug}` だけ**。SPA は hydrate して**二度と取り直さない**ので **`defaultLayout` / `permalinkPattern` が永久に undefined** → `resolveLayout` が `standard` に落ちる。**DB で `default_layout=bare` に直しても SPA に伝わっていなかった** |

- **SEO の方が重かった**: クローラは SPA を実行しないので、Google が見る `/privacy` は**汚染版が全て**だった。#536 の crawlable SSR が bespoke ページ全部にゴミを出し続けていた。
- 併せて **layout 未確定の間はテーマ chrome を推測描画しない**ように（推測は bare サイトで必ず外れる。**間違ったレイアウトは「壊れたサイト」に見え、空白は「読み込み中」に見える**）。

### C. #885 / PR #886 — bespoke ページが SPA 遷移になった（案A）

- 公開シェルに**委譲クリックリスナ**。判断は `resolveSpaLink` の**純関数**に切り出し、**21テスト**で除外条件（外部・PDF・mailto・修飾キー・中クリック・base path・ハッシュ）を固定。AYANE 全12ページの**実リンクを数えて**設計（内部239 / 外部78 / PDF1 / mailto1）。

## ⚠️ デプロイの前提（次回必読）

**本番の NENE2 は sibling ディレクトリ（本番ホストの sibling ディレクトリ）で `@dev` 解決**。07-15 時点で **7/8 相当（src 209ファイル）まで後れており、そのまま main を載せると #869 の `enableAuthorizationHeaderFallback` が古い NENE2 に当たり起動不能**だった（**デプロイ前チェックで捕捉**。ayane/aozora/apex が全部落ちるところだった）。

- **CI は `git clone --depth=1` ＝ NENE2 main を使う**。`composer.json` も `"hideyukimori/nene2": "@dev"`。
  → **v1.11.0 は「リリース済みだが誰も検証していない組み合わせ」で、main が実証済みの組み合わせ**。「released 版が安全」という一般則がこの構成では逆になる。
- `docker/php/Dockerfile.prod` は `additional_contexts: nene2` → `COPY --from=nene2 . /var/www/NENE2`。**`NENE2/` を先に rsync しないとビルドしても古いまま**。
- rsync `--delete` は NENE2 の `frontend/`（デモ用React）22件を消すだけで `src/` は無傷だが、**`--delete` は使わない**運用にした。

## ★教訓

1. **SPA マウント後の挙動は curl では見えない。** #880 で「SSR から chrome が消えた」ことを curl で確認して「直った」と報告したが、実機では出続けていた。
2. **`addInitScript` 内の `MutationObserver` は `document.documentElement` 未生成で登録に失敗し「0回」と誤検出する。** #881 の検証がこれで、2度目の誤報告の原因。
3. **決め手は動画のフレーム分解だった。** 「白 → Loading… → 本文」の実像と `p.searchhint` という DOM チェーンが取れて初めて、`/services` が1セグメントゆえ `PublicBrowsePage` に入るという**本当の経路**に辿り着けた。
4. **hydrate して refetch しない payload は、API と同形でなければならない。** #778 / #816 / #887 と3度繰り返した。**個別に気づいて足す運用は破綻している**ので機械で pin した。
