# 固定ページ階層化 ペルソナ会議（2026-06-27）

> 「WordPress 流の親子ページ（ネストURL）を採用すべきか」を WP 移行の市場ペルソナ4名（独立評価）で議論。現状＝コンテンツはフラット slug＋per-type permalink パターン＋メニュー(ナビ)ビルダーはあるが、**ページ階層（親子/ネストURL/parent picker/ツリー由来パンくず/子一覧）は無い**。

## 判定（4名）

| ペルソナ | 判定 |
| --- | --- |
| 制作会社（チャネル） | **adopt now＝table-stakes**（移行ブロッカー）。代替容認＝slug に `/`＋parent＋**リダイレクト取込（非交渉）** |
| コーポレート運用者 | **adopt now＝必須**。最低＝任意ネスト permalink＋**取込時 301 マップ**で旧URL 1:1 |
| 非技術ブロガー（大衆） | **不要／作るなら"既定で不可視"**。親子ピッカーは大衆を固まらせ「WordPressより簡単」を殺す |
| 移行/IA/SEO 専門家 | **ツリーは phase-later、代替を adopt now**。URL階層は順位に効かない(Google)・効くのは BreadcrumbList。ツリー自体が footgun |

## 収束点
- **ニーズは本物・収益直結**：ネストURLを忠実移行できないと corporate-rebuild 市場（＝有料ICP＝代理店チャネル）から締め出され、blog/LP ツールに typecast される＝**移行フィデリティは table-stakes**。
- **だが必要なのは "parent_id ツリー" でなく "URL 忠実性＋パンくず"**。"adopt" 派2人すら代替案（slug に `/` 許可＋リダイレクト取込）で妥協明言。IA/SEO が「ツリーは作るな、代替を作れ」。
- **ツリーの罠**：サブツリー移動＝子孫URL全変更の 301 雪崩・parent スコープ slug 衝突・循環防止・深さ/再帰クエリ＝移行が避けたい大量破壊そのもの。
- **大衆**：何を作るにせよ**既定で不可視**（複雑さは opt-in）。

## ★ 追加論点（ユーザー指摘）：ツリーは"管理側の整理"も担う
WP ツリーは二役＝①公開URL/IA ②**管理の整理・発見性**（ページ増で"ディレクトリ的に探す"）。①は permalink 代替でカバーだが②は未解決。
- **解＝permalink パスから"導出"**（parent_id 不要）：管理画面はパスでディレクトリ表示（折りたたみ・接頭辞グルーピング）＋一覧の検索/絞り込み/並べ替え；公開はセクション親が子を一覧＋パンくず。移動＝1レコードのパス編集（＋301）で雪崩なし。
- **既に現実の問題**：aozora 投入で1テナント1114レコード＝フラット一覧では探せない。「多数レコードの発見性（検索/絞り込み/グルーピング/仮想化）」は pages 抜きでも要る汎用課題。

## 決定（推奨）
- **WordPress の parent_id 親子ツリーは不採用。**
- 代わりに採用（80/20・4/4 収束）：
  1. **ページ単位の任意 permalink**（`/` を含むフリーパス・一意検証・スラッシュ正規化）。
  2. **バルク 301 リダイレクト取込（CSV）**＝移行の非交渉点（基盤：`url_redirects` テーブル #708＋permalink 変更時の自動301 が一部既存）。
  3. **BreadcrumbList JSON-LD**（パス由来）＋パンくず表示。
  4. **permalink パスを"導出階層"に**：公開＝子一覧/パンくず、管理＝ディレクトリ表示＋スケールする一覧。
  5. 大衆には**不可視/上級扱い**（既定 `/{type}/{slug}`）。URL 非依存の整理が要れば将来 "folder/collection" フィールド（任意）。
- **実装は別途**（本会議は判断と記録）。
