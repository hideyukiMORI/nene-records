# テナント URL 戦略ペルソナ討論 — subdomain vs ディレクトリ方式（2026-06-26）

> 論点：マルチテナント登録制 SaaS（`nene-records.com`）で、テナントを **subdomain**（`shop.nene-records.com`）と **ディレクトリ**（`nene-records.com/shop`）のどちらで分離するか。オーナー提起の3問を軸に討論：
> 1. テナントが `nene-suite` / `ayane.co.jp` 等へ **footer "powered by" リンク**を張るとき、subdomain 群からと単一ドメインからのどちらが価値が高いか。
> 2. **1ドメインとしてディレクトリ方式の方がドメイン価値（被リンク資産・権威）が高くならないか。**
> 3. 上記を踏まえた採るべき方式。

## 前提（実装・技術の事実確認）
- **両モードとも実装済**：`tenant_resolution_mode ∈ {single, subdomain, path}`。`SubdomainResolutionStrategy`（`org1.base → org1`）／`PathPrefixResolutionStrategy`（`/org1/... → org1`、middleware が prefix を strip）。
- **ただしディレクトリ方式は "公開SEOサイト" としては未完**（討論の重要前提）：
  - **(a) ルート衝突**：`PathPrefixResolutionStrategy` は `parts[0]` を無条件に org slug とみなす。公開 permalink（`/posts/1`・`/search`・`/{entityType}`）は全て `parts[0]` を使うので、`nene-records.com/shop/posts/1` の `shop` が org なのか entityType なのかが衝突する。**全テナント URL が必ず `/shop/...` で始まる規約**を強制すれば解けるが、現状そうは出来ていない。
  - **(b) URL生成が per-tenant prefix を吐かない**：今回の base path 基盤（#595/#596）は **固定の `APP_BASE_PATH`** を前置するだけで、**リクエスト毎に解決した org slug は前置しない**。よって canonical / sitemap / permalink / og:url がテナントの `/shop` を含まない＝**ディレクトリ方式の公開SEOは追加実装が必要**。
  - subdomain 方式は base path 不要（各テナントが subdomain 直下＝root）＝SSR/SEO はそのまま成立。ただしワイルドカード TLS が要る（VPS 必須）。

---

## ラウンド1 — 各ペルソナの初期ポジション

### SEO/テクニカルSEO専門家 → **「"powered by 被リンク価値" はほぼ幻想。そこで方式を決めるな」**
- Google は footer の**サイトワイド boilerplate リンクを強く割引**する。全テナントが同一テンプレで `nene-suite` へ張る "powered by" は、**1ドメインから来ようが subdomain 群から来ようが、ドメイン単位で名寄せされ実質1票**に圧縮される。規模が出ると逆に**リンクスキーム（不自然な大量同一アンカー）**と見なされ得る。
- 「subdomain 群なら被リンク元が分散して価値が上がる」は**誤り寄り**：Google は subdomain を親ドメイン配下に名寄せして評価する傾向が強い。`shop.nene-records.com` 100個からのリンクは `nene-records.com` 1ドメインからのリンクに近く扱われる。
- **本当に価値が出る被リンクは "テナントの独自ドメイン（`shop.com`）からの "powered by"** ＝真に別ドメイン。subdomain でもディレクトリでも `nene-records.com` 内に閉じている限り、被リンク多様性は出ない。
- 結論：**"powered by" の価値は SEO（リンク資産）ではなく "ブランド/リファラル流入（人間のクリック）"**。SEO を理由に方式を決めるのは筋が悪い。

### コンテンツ/グロースマーケ → **「ドメイン権威の集約はディレクトリが正しい。オーナーの直感は当たっている」**
- 質問2はその通り：**ディレクトリ方式は全テナントのコンテンツと被リンクを `nene-records.com` 1ドメインに集約**する。ページ数・トピック網羅・流入リンクが1ドメインに積み上がり、**ドメイン権威（DA 的なもの）は確実にディレクトリが高くなる**。「subdirectory に移したら順位が上がった」事例は SEO 界で多数。
- subdomain は権威を**各 subdomain に分割**し、root は痩せる。「強い1ドメインを育てる」戦略ならディレクトリ。
- **ただし諸刃**：低品質テナントが増えると**ドメイン全体の評価を引き下げる**（thin-content / doorway farm 判定リスク）。1ドメインに乗せる＝**運命共同体**。

### 同業プラットフォーム観察者（前例）→ **「WP.com・Substack・Shopify・Tumblr は全部 subdomain ＋独自ドメイン。理由がある」**
- 大手テナント型は**例外なく `user.platform.com` の subdomain ＋ 本気勢は独自ドメイン**：
  - WordPress.com（`you.wordpress.com`）/ Substack（`you.substack.com`）/ Shopify（`you.myshopify.com`）/ Tumblr / Netlify / Vercel。
- 採用理由＝**(1) スパム隔離（1テナントの低品質/規約違反が母ドメインを汚さない）、(2) テナントの独立性（"自分のサイト" 感）、(3) 独自ドメインへの移行が自然**。
- **ディレクトリ方式を大規模登録制でやっている主要プレイヤーは事実上いない**。これは強いシグナル＝**ディレクトリ集約の "ドメイン権威" メリットより、隔離・独立性のほうが実運用で重視されてきた**。

### プラットフォーム事業戦略（運営=nene-records.com の価値最大化）→ **「ドメイン価値は欲しい。でも "テナントを下層に飼う" 形で取りに行くのは危険」**
- `nene-records.com` を**権威あるドメインに育てたい**のは事業として正しい。ただしそれを**テナントコンテンツの寄せ集めで稼ぐ**と、品質コントロール不能＋ブランド毀損リスクを母体に直結させる。
- **代替**：ドメイン価値は**運営自身の一次コンテンツ（事例集・テンプレギャラリー・ブログ・ドキュメント）＋ "discover/showcase" 的キュレーション hub**（`nene-records.com/discover`）で積む。テナント本体は subdomain/独自ドメインに置き、hub から **審査済みの優良テナントだけ**にリンク。**集約のメリットを品質管理付きで取り、スパム共同体リスクを切り離す**。

### テナント当事者（中小EC/店舗オーナー）→ **「`nene-records.com/myshop` は "間借り" に見える。`myshop.〜` か独自ドメインがいい」**
- 顧客に見せる URL が `nene-records.com/myshop` だと**他社サイトの下層ページ**＝自社ブランドに見えない。名刺・チラシに書きにくい。
- `myshop.nene-records.com` なら "自分のサイト" 感が出る。**いずれ独自ドメイン（`myshop.com`）に移せる**前提が欲しい（subdomain なら CNAME で自然、ディレクトリからの移行は 301 地獄）。
- "powered by" は**付けてもいいが、自分のブランドを食わない範囲**で。むしろ**独自ドメインに移れるか**が決め手。

### インフラ/アーキ → **「現状ディレクトリ方式の公開SEOは未配線＋ルート衝突。subdomain の方が技術的に素直」**
- 上記前提(a)(b)の通り：ディレクトリ公開サイトは **per-tenant slug を URL 生成に通す改修＋ `parts[0]` 衝突解消（全公開ルートを `/{org}/...` 規約に寄せる）** が要る。**permalink 設計の作り直しに近い**。
- subdomain は **base path 不要・SSR/SEO そのまま・cookie origin 分離が自然**。コストは**ワイルドカード TLS（VPS＋DNS-01）**のみ。
- **技術的負債の総量は subdomain < ディレクトリ**。

### セキュリティ/法務 → **「同一オリジン共有は事故時の被害半径が大きい。隔離は subdomain」**
- ディレクトリ＝全テナント同一オリジン。XSS（カスタムページ/バンドル）・cookie・CSP の**被害半径がドメイン全体**。バックエンドの `org_id` 照合で越境 "アクセス" は塞げるが、**ブラウザ実行コンテキストの隔離は別問題**。
- 規約違反/法的削除要請が来たとき、subdomain なら**その subdomain だけ**を止められる。ディレクトリは母ドメインの評判に直結。

### 収益/事業（連続起業家）→ **「独自ドメインこそ収益とSEOの両取り。方式はその通り道で選べ」**
- 価値ある被リンクも、テナントの満足も、**独自ドメイン（`shop.com`）に集約**される。だから戦略は「**無料は subdomain → 有料で独自ドメイン**」（Shopify/Substack モデル）。
- これは**収益SKUと完全に噛み合う**（独自ドメイン＝有料機能）。ディレクトリ方式は独自ドメイン導線が不自然で、**マネタイズの階段を1段失う**。

## ラウンド2 — 反論と収束

- **グロースマーケ（ディレクトリ擁護）への反論**：SEO専門家＋観察者が「集約のドメイン権威は事実だが、**(1) スパム共同体リスク、(2) テナント独立性の喪失、(3) 独自ドメイン移行の断絶**を母体に背負う対価が高すぎる。**権威集約は "運営の一次コンテンツ＋審査済み hub" で代替取得できる**（事業戦略ペルソナ案）」。グロースマーケ、**「集約は hub で取る」案に合流**。
- **"powered by 被リンク" 問題（質問1）の決着**：SEO専門家の整理に全員合意。**footer boilerplate リンクは SEO 的にはほぼ無価値（むしろ大量だとリスク）**。subdomain vs ディレクトリで被リンク価値は**有意差なし**（どちらも `nene-records.com` 内に名寄せ）。**"powered by" の真価はブランド/リファラル流入**＝ならば**人がクリックしたくなる体験**（速い・綺麗・"このサイトいいな→何で作った？"）が本質で、URL構造の問題ではない。**真に価値ある被リンクは独自ドメインのテナントから**。
- **質問2（ドメイン価値）の決着**：オーナーの直感「ディレクトリの方がドメイン価値が高くなる」は **SEO 機序としては正しい**。ただし**それを取りに行く手段としてテナントを下層に飼うのは割に合わない**。**同じドメイン価値を、運営一次コンテンツ＋審査制 showcase hub で、スパムリスクなしに取る**のが上位互換。
- **テナント当事者＋収益＋セキュリティ＋インフラ**は最初から subdomain 寄りで不動。観察者の "業界全員 subdomain＋独自ドメイン" が決定打。

## 判定分布
| ペルソナ | 方式 |
| --- | --- |
| SEO専門家 | subdomain（被リンクは方式無関係・独自ドメイン推し） |
| グロースマーケ | **ディレクトリ→ hub 案へ転向** |
| 前例観察者 | subdomain＋独自ドメイン |
| 事業戦略 | subdomain ＋ 運営hubでドメイン価値 |
| テナント当事者 | subdomain/独自ドメイン |
| インフラ/アーキ | subdomain |
| セキュリティ/法務 | subdomain |
| 収益/事業 | subdomain→独自ドメイン（SKU） |

**8/8 が subdomain（＋独自ドメイン）に収束。ディレクトリ単独支持はゼロ。**

## 決定的洞察

1. **質問1（"powered by" 被リンク）は前提が崩れている。** footer boilerplate リンクは SEO 的にほぼ無価値で、subdomain でもディレクトリでも `nene-records.com` に名寄せされ差は出ない。**価値はリンク資産ではなくブランド/リファラル流入**。SEO で意味ある被リンクが欲しいなら **"テナントを独自ドメインに乗せる"** のが唯一効く道（別ドメイン＝真の多様性）。

2. **質問2（ドメイン価値集約）はオーナーの直感が機序として正しい**。ディレクトリは権威を `nene-records.com` に集約する。**だが手段が問題**：テナントコンテンツの寄せ集めで取りに行くと、**スパム共同体リスク・テナント独立性喪失・独自ドメイン移行の断絶**を母ドメインに背負う。

3. **上位互換の結論＝"分離は subdomain、ドメイン価値は運営hub"**：
   - **テナント本体は subdomain**（無料）**→ 独自ドメイン**（有料SKU）。隔離・独立性・収益階段・SSR/SEO そのまま成立。業界標準と一致。
   - **`nene-records.com` のドメイン価値は、運営の一次コンテンツ（事例・テンプレギャラリー・ブログ・docs）＋審査制 showcase/discover hub** で積む。集約メリットを**品質管理付き**で取り、スパムリスクを切り離す。
   - **"powered by" は SEO 目的でなくブランド導線として設計**（nofollow 前提でも可）。効くのは "見た人が作りたくなる体験"。

4. **技術裏付けも subdomain 有利**：ディレクトリ公開SEOは未配線（per-tenant prefix を URL 生成に通す改修＋`parts[0]` 衝突解消＝permalink 作り直し級）。subdomain は base path 不要・SSR/SEO 即成立・origin 分離が自然で、コストは**ワイルドカード TLS（ConoHa VPS＋DNS-01）**のみ。→ **Heteml では張れない＝ConoHa が必要**という前回の結論とも整合。

## 結論（討論の総意）
- **テナント分離は subdomain（→独自ドメイン）一択。ディレクトリ単独方式は採らない。**
- **`nene-records.com` のドメイン価値は "テナントを下層に飼う" のではなく "運営の一次コンテンツ＋審査制 hub" で築く。**
- "powered by" は**ブランド/リファラル施策**として設計（SEO リンク資産は期待しない）。SEO 被リンクが欲しければ**テナントの独自ドメイン化**を促す。
- ホスティングは**ConoHa VPS（ワイルドカード TLS）**。これは前回の subdomain 結論と二重に整合。

## 次の一手（事業判断はオーナー）
1. テナント解決＝**subdomain モード固定**で managed signup 最小スライス（公開 signup→org自動プロビジョン→`slug.nene-records.com` 割当→即ログイン）。
2. 有料SKUに**独自ドメイン接続（CNAME＋自動TLS）**を据える。
3. `nene-records.com` ルートは**運営の showcase/discover hub ＋一次コンテンツ**として設計（ドメイン価値の正攻法）。

## 追記（2026-06-26・オーナー指摘で補正）：ディレクトリモードは "不要" ではない

討論の「subdomain 一択」結論は **`nene-records.com` が運営する登録制 SaaS**（＝運営がホストを握る＝ConoHa＝ワイルドカード TLS 可）に**スコープ限定**。製品全体から**ディレクトリモードを外す意味ではない**。

- **別ユースケース＝自前ホストのマルチテナント**：zip/MIT の自己ホスト層には、**ワイルドカード subdomain TLS を張れない共有レンタルサーバ**（Heteml 等）で複数テナントを運用したい層がいる（代理店・リセラー・複数サイト運用、社内の部門別サイト等）。**"利用者全員が subdomain を自由に張れる層" とは限らない**。彼らには**ディレクトリが唯一の現実解**。`PathPrefixResolutionStrategy` のコメント「Best for shared-host deployments where wildcard subdomains are not available」が示す通り、これは**意図された fallback**。
- **product stance＝3モード全対応の柔軟性こそ価値**：`single`（base path で単一サイト）／`subdomain`（VPS・推奨 for 自社SaaS）／`path`（制約ホストのマルチテナント fallback）を**デプロイ環境の制約で選べる**のが製品の強み（WordPress が単一なのに対する差別化にもなる）。
- **現状の到達点**：ディレクトリは **API/admin のテナント解決は動く / 公開SEOサイトとしては未完**。完成には (a) `parts[0]` 衝突解消（全公開ルートを `/{org}/...` 規約へ寄せる）＋ (b) URL 生成に per-tenant slug を前置（base path 機構の "per-request org" 版）が要る。
- **優先度**：自社 SaaS（subdomain）が先。**ディレクトリ公開SEO完成は "自前ホスト・マルチテナント" の実需が出た時**（or zip インストーラの "上級" オプションとして）。実装は**残す**が、公開SEO配線は**需要ドリブン**で。

**修正結論**：「ディレクトリ削除」ではなく → **「`nene-records.com` 自社SaaS は subdomain を選ぶ。ディレクトリは "制約ホストでの自前マルチテナント" の fallback として保持し、公開SEO配線は実需が出たら完成させる」**。3モードを環境制約で選べる柔軟性そのものが製品価値。
