## 前提：独自ドメインの技術的現実（answer 合わせ）

収益SKUの中核候補＝独自ドメインの実装コストを正しく見積もるため、現状をコードで確認した。

- ✅ **DNS → Caddy**：顧客が独自ドメインを Caddy ホストへ向ける（A / CNAME）。
- ✅ **on-demand TLS（`ask` = `TlsCheckHandler`）は既に custom domain を許可済み**：base 配下でない host は `findByCustomDomain($host)` で照合し、active org にヒットすれば 200 を返して証明書を自動発行。**「登録すれば証明書は出る」は実装済み**（#600 の派生）。
- ⚠️ **穴1：org 解決の配線**。現行 subdomain モードの active strategy = `SubdomainResolutionStrategy` は `*.nene-records.com` 以外を **null → 404** にする。`CustomDomainResolutionStrategy`（full host → `findByCustomDomain`）と repo メソッドは**存在するが active path に組み込まれていない**。→ 証明書は出るが中身が 404。修正は小（subdomain で外れたら custom domain にフォールバックする複合 strategy、または OrgResolver の null 分岐で `findByCustomDomain(fullHost)`）。
- ⚠️ **穴2：テナント自助登録**。`custom_domain` を設定できるのは現状 **superadmin の組織 CRUD のみ**。テナント自身の「ドメイン追加」UI と所有権検証（DNS TXT 等）は無い。有料SKU化は (a) 当面**手動登録＝コンシェルジュ**、(b) 後で自助＋検証。

**結論**：TLS という難所が済んでいるため、独自ドメインSKUの engineering は**小さい**。最初の有料機能に適する。

`organizations.plan` の enum（`free/starter/pro/enterprise`）は既存だが**完全な空箱（どこも enforce していない）**。ここに意味を与える。数値はすべて**仮（soak で検証）**。

### SKU / add-on（ティア横断）
- **独自ドメイン**：Starter 以上に同梱（Free は不可）＝最初の課金トリガー。
- **保守SLA / サポート契約**：Enterprise の核（SaaS営業・中小経営の決め手）。**zip 自己ホスト勢向けに単体「保守プラン」**を別軸で（Priya 対応）。
- **(将来) AI / MCP 使用量メータリング**：AI-native の差別化を従量課金で獲る余地。

### 課金メカニズム（段階）
- **フェーズ1（今すぐ・soak 並行）＝コンシェルジュ**：手動請求＋superadmin で `plan`/`custom_domain` を設定。最初の数社で**支払い意思と価格を検証**。engineering ほぼゼロ。
- **フェーズ2＝Stripe サブスク**：Free→Starter/Pro の自助課金。Enterprise は引き続き見積/請求書。

### Free / 有料の線引き
Free は本当に使える（signup・soak のドライバ）が、**商用に要るもの（独自ドメイン・ブランド除去・サポート・スケール・SSO）は有料**＝「個人/お試しは無料、商売は有料」。

### engineering 実装順（コスト感）
1. **plan ゲート層**（空箱に魂）— まず**独自ドメイン可否＋ブランド除去フラグだけ**（小）。上限は soft 警告→hard を段階的に。
2. **独自ドメイン**：resolver フォールバック配線（小）。自助登録＋検証は後（中）。コンシェルジュなら今ゼロ。
3. **課金**：コンシェルジュ＝ゼロ／Stripe＝後（中）。
4. **2FA / SSO** は #543 に相乗り（Pro/Ent）。

## 決定的洞察
1. **"実装したもの"だけが人を動かした**：今回動いた1名は提案ではなく**シップ済みの rate-limit**。収益モデルは提案段階ゆえ誰も今すぐに飛ばなかった。**第6回の教訓（"製品体験として完結しているか"が閾値）の再確認**＝紙のSKUは閾値を越えない。
2. **しかし収益ゲートの性質は縮んだ**：3名の条件は「収益導線が存在しない」→「**価格/SLA/保守窓口を決めて検証する**」へ。**ブロッカーが"未定義"から"未実行"に降りた**＝次の一手が明確になった。
3. **収益は依然 soak の後**：SaaS営業の「参照顧客」、連続起業家の「価格検証」、中小経営の「先行事例」はいずれも **soak の実テナント蓄積**に依存。第6回の処方順（**(C)堅牢化 → (B)soak → (A)収益**）は本回でも妥当。

## 次の一手
1. **(C) 仕上げ**：独自ドメインの resolver 配線（小）＝SKU を売れる状態の前提。rate-limit は本番反映を確認。
2. **(B) soak 開始**：実テナント/実コンテンツを投入し数週間運用（James/陳偉/SaaS営業の"実績"を時間が解く）。
3. **(A) 収益の事業判断**：soak と並行して **2ティア（Free/Pro）＋コンシェルジュ課金＋ゲートは独自ドメイン＆ブランド除去だけ**で最小開始 → 最初の有料客で価格・SLA を検証 → Stripe 自助と上限ゲートを後追い。**enforce してから広告**。

## 結論
**収益モデルの"提案"は、収益ゲート3名の条件を「不在」から「検証待ち」へ縮小したが、提案である限り誰も今すぐには動かさない。** 実際に動いたのは**シップ済みの rate-limit（#613）**で見送りが2→1 に減った1名のみ。これは「実装が完結したものだけが閾値を越える」という第6回の教訓の再確認であり、**収益は soak で実顧客を得てから価格・SLA・保守窓口を確定・実装する**のが筋であることを裏づける。直近は **独自ドメイン配線（小）→ soak → コンシェルジュ課金での価格検証** の順。
