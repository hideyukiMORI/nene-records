# 経営ペルソナ会議 — 次の sequencing（2026-06-26）

> ADR 0005 系（収益/拡張）が本番まで到達した後、「次に何を/どの順で」を経営6ペルソナ（独立評価）に当てさせた回。
> 選択肢を中立に提示：**A=zip インストーラ / B=soak / C=③ self-serve 自動化（Stripe＋自助ドメイン）/ D=コンシェルジュ継続＋事業判断（価格・初の有料客）**。
> 前提：managed SaaS は本番稼働・PlanBased も本番で有効・/account 閲覧面あり・**有料客 ≈ 0**・課金はコンシェルジュ（Payment Link＋手動 plan）で**今日から可能**（決済はブロッカーでない）。

## 判定（各自の順序）

| ペルソナ | 順序 |
| --- | --- |
| OSS創業者 | D+B 今 → A 次 → C 後 |
| 個人開発者 | D 今 / B 並行 → C 後 → A やらない |
| VC | D+B 今 → A 次 → C やらない |
| 連続起業家 | D 今 / B 並行 → C 後 → A やらない |
| CFO/法務 | D+B 今 → C 後 → A やらない |
| パートナーシップ | D+B 今 → A 後 → C やらない |

## 全会一致（強い）
1. **今＝D＋B（並行）**。D＝価格/SKU を決め、**コンシェルジュで有料客#1を取りに行く（ビルドでなく営業）**。B＝soak（実テナント/実コンテンツ投入・数週放置・安い・参照と retention を生む）。6/6。
2. **C（③ self-serve 自動課金）＝後回し**。コンシェルジュが実際に痛くなってから（目安：有料 ~10／請求運用が週半日超／"今すぐ自分で払って設定したい"で失注）。6/6。
3. **A（zip）＝"最初"ではない**（D+B の後）。

## 割れた点＝A（zip）の重み
- **やらない派（個人/連続/CFO）**：self-hoster は払わない＝$0・支援負担・投機。
- **次にやる派（OSS/VC/パートナーシップ）**：zip こそ WP 規模の本命 moat／流通エンジン。near-zero dev cost で**ビルドは安い**——真の制約は**支援負担・用地の窓・実績/需要**。だから「証拠が出たら即」。

## 決定的インサイト
- **near-zero dev cost は罠**：作るのが安い→過剰建設。手動が痛くなってから自動化/建設。
- **soak は隠れ蓑になりうる**：磨きは"生産的に見える"ので、怖い部分（人に金を請う）を避けがち。**D は営業時間として死守**。
- **真の未知は"転換 vs リーチ"**：D は「払うか(転換)」を試す。規律ある営業でも無 audience で売れない（リーチ問題）なら、**A（無料 self-host を CAC/流通エンジンに）を昇格**＝A はヘッジ。
- **A を"never"にしない**：放置すると AI-native ライバル/WP に self-host beachhead を取られ、"こぢんまり managed lifestyle"で大きな outcome を逃す。
- **顧客#1を"代理店"に**（パートナーシップ）：1契約で下流多数＋参照。

## 統合結論（sequencing）
- **今＝D＋B**：価格/SKU を決め、（理想は代理店を）顧客#1にコンシェルジュで売る＋soak 開始。
- **安い布石を今**：**PHP 8.4 gate を PoC で潰す**（A の唯一の技術前提・数時間）→ self-host pull が来たら A を即発火できる状態に（建設はしない）。
- **シグナル待ち**：self-host pull（GitHub star／"どう入れる?"／代理店の self-host 希望）→ **A 昇格**。コンシェルジュが痛い（~10有料 等）→ **C 昇格**。

→ 「zip が ③ より先か？」の答え：**③(=C) を後回しにする判断は正しい。zip(A) は ③ より価値が高い"次の大物"だが"最初"ではない——まず売る(D)＋soak(B)。zip は PHP8.4 gate だけ今潰し、self-host pull が出たら発火。**
