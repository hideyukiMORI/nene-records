## 課題・所感

**コンフリクト解消の連鎖**  
M5 の PR 2 本が同じ `AppShell.tsx` / `router.tsx` / `i18n` ファイルを触っていたため、マージするたびに次がコンフリクト。「両方の変更を union で残す」という原則を守ることで安全に解消できた。

**エラーパターンの繰り返しを防ぐ参照確認の重要性**  
コメント実装で発生したバグ 3 件（インターフェース名・OpenAPI $ref・ボディパーサー）は、いずれも既存の類似コード（`NavigationItemNotFoundExceptionHandler` / users エンドポイントの inline 401 / `CreateNavigationItemHandler`）を最初に参照していれば防げた。実装前に同系統の既存コードを確認するルーティンを徹底したい。

**フルスタック一気通貫の達成感**  
コメント機能は DB マイグレーションから OpenAPI 仕様、PHP ドメイン層、React エンティティ層・フィーチャー層・ページ層まで一人で完結させた。NENE2 パターンと FSD が軌道に乗っているため、新機能の「型」が掴めている感覚がある。

## 課題・所感

**「検証カバレッジを積み上げた後にレビューする」サイクルの価値**  
M10 でテストを一気に 217 本まで伸ばした後、全体をレビューして Issue 化するというサイクルは効果的だった。テストが揃っているおかげで、「これを直しても既存の動作は崩れない」という安心感を持って Issue を書ける。

**Issue を積むことの価値**  
「今は壊れていない」問題を見つけたとき、その場で直すか Issue に積むか判断する必要がある。今回は 5 件すべてを Issue に積んだ。理由は、緊急度が低く、かつ変更範囲が複数ファイルにまたがるため、単独セッションでクリーンに直すに値する。緊急度の高い bug は別 Issue（#271、#268）として既に存在しており、優先順位の判断は正確にできている。

**`docs/claude-impressions.md` という記録**  
3 日前の自分（Claude）が書いた懸念点が全部解消されていた。この「AI が書いたレビューを、同じ AI が 3 日後に参照して更新する」ループは、人間チームが週次スプリントでレトロスペクティブを行うのと構造的に同じことをしている。
